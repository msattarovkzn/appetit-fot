# PROJECT_STATUS — «Аппетит — ФОТ»

> Срез состояния проекта на 2026-06-08. Источники: `PROJECT_DOCUMENTATION.md` (READ-ONLY аудит) + изменения, внесённые после него (см. `CHANGELOG.md`).

---

## 1. Что реализовано

- **Аутентификация**: вход по логину/паролю (`/auth/login`, JWT), проверка PIN сотрудника, смена пароля
- **Открытие/закрытие смен по PIN** (`/shifts/*`) — обычный флоу + защита от двойного открытия
- **Доп. смены через кассира** (`/cashier/extra-shift/*`)
- **Закрытие дня кассиром по PIN** (`/cashier/close-day-by-pin`) — основной рабочий флоу: выручка → `BranchDailyReport` → пересчёт ФОТ → Telegram-уведомление
- **Расчёт ФОТ** (`calculate_payroll_for_day`) — почасовая/фиксированная оплата, бонус кассира, статусы по категориям
- **Бухгалтерская проверка дней** (`/review/*`) — статусы red/yellow/green, корректировки смен, аннулирование, аудит-лог
- **Управление сотрудниками/должностями/ставками** (`/admin/*`) — актуальный путь, с проверкой уникальности PIN и автозакрытием предыдущих ставок
- **Дашборды и аналитика** (`/dashboard/*`, `/analytics/*`) — сводки по ФОТ, тренды, сравнение периодов
- **Планирование графика** (`/schedule/*`) — план/факт по неделям
- **Публичный монитор смен** (`/shifts/live`) — экран "кто сейчас на смене" по всем филиалам, без авторизации, с замаскированными ФИО (с 2026-06-08)
- **Кабинет сотрудника** (`/employee/*`) — read-only просмотр своих смен/начислений/графика, отдельная JWT-система
- **Единый модуль бизнес-правил** (`app/business_rules.py`) — пороги ФОТ и формула бонуса кассира, извлечены из 5 дублирующихся копий (коммит `1f7b778`, 2026-06-08)

## 2. Что реализовано частично / спорно

1. **Два параллельных аудит-механизма**:
   - `app/services/audit.py::log_action` — легаси, был несовместим со схемой БД (передавал несуществующее поле `ip_address`); **исправлено** коммитом `fa0ccfb` (2026-06-08), но сам механизм остаётся "вторым" наряду с актуальным
   - `app/utils/review_helpers.py::write_audit` — актуальный, используется в `admin.py`/`review.py`
   - Решение по объединению/устранению дублирования audit-механизмов **не принято**
2. **Дублирование роутеров `/employees` vs `/admin/employees`** — частично устранено: legacy write/rate-эндпоинты в `employees.py` заменены на `410 Gone` (коммит `fcbd8a6`); `GET /employees/` оставлен рабочим (используется фронтендом). Сама дупликация модели/таблицы осталась, но риск опасных вызовов снят
3. **`Shift.approved_by`** — поле существует, но не используется по прямому назначению («кто утвердил смену»); в `cashier.py:open_extra_shift` явно передаётся `None`
4. **`takeaway_count`** — собирается по всей цепочке (отчёты, аналитика), но не участвует в расчёте ФОТ; неочевидно для пользователей UI, что поле информационное
5. **`test_mode.py`** жёстко привязан к `branch_id=1`, активность зависит от `ENVIRONMENT` — требуется регулярная проверка, что в проде `ENVIRONMENT=production`
6. **Категория должности `reserve`** присутствует в формуле ФОТ и seed-данных, но неясно, есть ли реальные сотрудники этой категории

## 3. Открытые задачи

1. **Расследование PIN-проблемы Ямашева (Казань, branch_id=2)** — пользователь сообщил, что PIN 1001 "не работает" для сотрудника этого филиала. Расследование **заблокировано** — пользователь отклонил оба предложенных подхода (live-тест API и read-only DB-запрос). Требуется новый метод проверки по согласованию с пользователем
2. **`review.py` — magic numbers в `day_verdict()`** (строки ~255, ~258): хардкод `29.0`/`15.5` вместо констант из `business_rules.py`; не исправлено в рамках рефакторинга `business_rules`, т.к. `total_fot_pct`/`kitchen_fot_pct` явно кастуются в `float`, а константы — `Decimal` (риск `TypeError` при наивной замене). Вынесено отдельной фоновой задачей (`task_6dc56a98`)
3. **Push в `main` после фикса таймзоны** (коммит `3ae590b`) — статус подтверждения push в удалённый репозиторий не зафиксирован финально в переписке (вопрос был задан, ответ переключился на другую тему)
