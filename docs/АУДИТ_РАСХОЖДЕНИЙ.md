## Раздел 2. Описано, но не реализовано (или реализовано частично)

### 2.1 is_test_data flag
**Описано:** В запросе на live mode упоминается флаг `is_test_data=false` для новых записей.
**Реализовано:** Флага нет ни в одной таблице. Нет миграции.
**Рекомендация:** Добавить в Migration 008 (если нужна граница тест/live). Альтернатива: фильтровать по дате — все данные до даты старта live считать тестовыми.

### 2.2 Пересечение смен (overlap detection)
**Описано:** "Запретить создание смен с пересечением по времени".
**Реализовано:** Проверка только на "открытая смена сегодня". Проверки пересечения по времени нет.
**Рекомендация:** Добавить в `/shifts/open` и `/cashier/extra-shift/open` проверку на overlap по `opened_at..closed_at`.

### 2.3 approved_by для extra_shift
**Описано:** Кассир, открывающий доп. смену, должен быть записан как `approved_by`.
**Реализовано:** `approved_by` — FK на `users.id`, но кассиры входят через PIN (не как Users). Поле записывается как `null`, информация о кассире хранится в `shift.note`.
**Рекомендация:** Либо принять как допустимо (note хранит имя кассира), либо добавить поле `approved_by_employee_id INT FK employees.id`.

### 2.4 Номер телефона как логин (/employee)
**Описано в концепции:** Логин через номер телефона.
**Реализовано:** Логин через `employee_login` (произвольная строка, задаётся бухгалтером) + PIN. Поле `phone` есть в модели, но не используется для авторизации.
**Рекомендация:** Текущая реализация проще и гибче. Телефон можно добавить как альтернативный логин позже.

### 2.5 Тестовый режим через ENVIRONMENT
**Описано:** `LIVE_TEST_MODE=true` — новые shifts создаются с `is_test_data=false`.
**Реализовано:** Флаг `is_test_data` не существует в моделях. `LIVE_TEST_MODE` влияет только на UI (скрывает тестовые контролы).
**Рекомендация:** Для production-разграничения добавить migration 008 с полем `is_test_data boolean default false` и ставить `true` для данных, созданных через `/test/*` endpoints.

## Раздел 3. Противоречия

### 3.1 Роль cashier
**В концепции:** Кассиры имеют роль `cashier` в системе.
**В коде:** Роль `cashier` есть в enum `UserRole`, но ни один реальный пользователь с этой ролью не создаётся. Кассиры аутентифицируются через employee PIN + флаг `is_cashier=true` в таблице employees.
**Решение принято:** Оставить текущую архитектуру. Кассир = сотрудник с `is_cashier=true`.

### 3.2 Менеджер (manager1) видит все филиалы
**В концепции:** Менеджер видит свой филиал (branch_id).
**В коде:** `manager1.branch_id = NULL` → видит все филиалы. Это было явно настроено по запросу.
**Решение принято:** NULL branch_id = все филиалы. Документировать как feature.

### 3.3 close-day-by-pin: блокировка повторного закрытия
**В исходном коде:** "День уже закрыт" блокировало второго кассира.
**Сейчас:** Каждый кассир закрывает свою сессию, BranchDailyReport суммируется. Блокировка снята.
**Решение принято:** Текущая реализация корректна для мультикассира.
