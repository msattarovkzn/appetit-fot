# PROJECT_SPEC — «Аппетит — ФОТ»

> Спецификация системы. Составлена на основе аудита (`PROJECT_DOCUMENTATION.md`, 2026-06-08) и текущего кода. Цель — дать новому участнику/чату понимание системы за 10-15 минут.

Стек: **FastAPI + async SQLAlchemy + PostgreSQL** (`backend/`), **Next.js 14 + TypeScript + TailwindCSS** (`frontend/`).
Деплой: фронтенд → Vercel (автодеплой из `main`), бэкенд+БД → Timeweb Cloud VPS (Docker Compose, ручной деплой).

---

## 1. Назначение системы

Учёт рабочих смен и автоматический расчёт ФОТ (фонда оплаты труда) для сети точек питания «Аппетит» — 5 филиалов (Челябинск + 4 точки в Казани: Ямашева, Глушко, Хади Такташ, Шакирова).

Базовый сценарий: сотрудники открывают/закрывают смены по PIN-коду на планшете точки → кассир в конце дня вводит выручку и закрывает день → система считает ФОТ по категориям должностей и шлёт уведомление в Telegram → бухгалтер проверяет/корректирует дни → управляющий смотрит аналитику и планирует график.

## 2. Роли

`UserRole`: `employee`, `cashier`, `manager`, `accountant`, `owner`.

Иерархия ролевых зависимостей (`backend/app/dependencies.py`):
```
require_cashier    = cashier, manager, accountant, owner
require_manager    = manager, accountant, owner
require_accountant = accountant, owner
require_owner      = owner
```

| Роль | Доступ |
|---|---|
| **owner** | полный доступ |
| **accountant (бухгалтер)** | управление сотрудниками/должностями/ставками (`/admin/*`), проверка дней (`/review/*`), корректировка payroll, аудит-лог; график — только чтение |
| **manager (управляющий)** | дашборды, аналитика, список смен, утверждение часов, график (запись); сотрудниками/ставками НЕ управляет |
| **cashier (кассир)** | реальный флоу закрытия дня — через PIN сотрудника `is_cashier=True`, без отдельного User-аккаунта |
| **employee (сотрудник)** | нет прямого доступа по роли User; работает через PIN (модель `Employee`) либо отдельный кабинет (`employee_login` + свой JWT `type=employee`, параллельная система токенов) |

## 3. Бизнес-процессы

### 3.1 Открытие смены
`backend/app/routers/shifts.py`
1. `POST /shifts/status` — UI проверяет, есть ли уже открытая смена сегодня (по PIN, без создания записи)
2. `POST /shifts/open` — поиск сотрудника по PIN (`_find_employee_by_pin`, перебор активных сотрудников филиала и проверка bcrypt-хеша); проверки:
   - нет уже открытой смены сегодня → иначе `400 "Смена уже открыта"`
   - не было закрытой смены сегодня → иначе `400 "already_had_shift"` (повторную/доп. смену может открыть **только кассир**)
   - создаётся `Shift(status=open, opened_at=now)`

### 3.2 Закрытие смены
3. `POST /shifts/close` — находит открытую смену сотрудника, ставит `closed_at=now`, `status=closed`, считает:
   - `total_minutes` = из таймстампов
   - `total_hours_decimal` = `round(total_seconds/3600, 4)`
   - `approved_hours` = `round(total_seconds/3600, 2)` (предварительное значение, может быть скорректировано бухгалтером позже)

### 3.3 Доп. смена через кассира
`backend/app/routers/cashier.py`
4. `POST /cashier/extra-shift/open` — кассир (PIN, `is_cashier=True`) открывает смену другому сотруднику; время — переданное или текущее, `is_extra_shift=True`
5. `POST /cashier/extra-shift/close` — кассир закрывает любую смену (свою или чужую), аналогичный расчёт времени

### 3.4 Закрытие дня (основной флоу)
`POST /cashier/close-day-by-pin` — единая операция, выполняемая кассиром по PIN:
1. Проверка, что кассир ещё не закрывал день сегодня
2. Закрывает свою личную смену (если открыта)
3. Считает бонус кассира (`cashier_bonus`: 7₽/заказ Пн-Чт+Вс, 5₽/заказ Пт-Сб — единая формула, см. `business_rules.py`)
4. Создаёт `CashierSession`; суммирует/создаёт `BranchDailyReport` (поддержка нескольких кассиров за день — выручка и заказы суммируются)
5. Запускает пересчёт ФОТ — `calculate_payroll_for_day`
6. Собирает список незакрытых смен филиала за день
7. Отправляет итоговое уведомление в Telegram

> Старый JWT-флоу (`GET /cashier/reports`, `POST /cashier/close-day`) — закрытие дня бухгалтером/менеджером — существует параллельно как альтернативный путь.

### 3.5 Закрытие дня — корректировки бухгалтером (после факта)
Два независимых пути:
- `PATCH /admin/shifts/{id}` — простая коррекция (`approved_hours`, `note`) → пересчёт `PayrollEntry`/`FotSummary`
- `PATCH /review/shifts/{id}/correct` — полная коррекция: время открытия/закрытия, прямое переопределение часов, override ставки, аннулирование смены (`annul=true`, требует комментарий)

### 3.6 Бухгалтерская проверка дня
`backend/app/routers/review.py`
- `GET /review` — список филиалов со статусом проверки дня: `red` / `yellow` / `green`
- `GET /review/{branch_id}/{date}` — детальная карточка дня: payroll, план/факт, текстовый вердикт (например «ФОТ превышен на N п.п.»)
- `POST /review/{branch_id}/{date}/verify` — закрыть проверку дня (статус → `green`)
- `POST /review/{branch_id}/{date}/reopen` — переоткрыть проверку (`green` → `yellow`)
- `POST /review/shifts/{id}/resolve` — разрешение аномалии (например, аномально длинная смена)
- `GET /review/audit-log` — журнал аудита изменений

Статус дня хранится в `daily_branch_review` (поля `status`, `issues` JSONB, `issues_count`, `reviewed_by/at`).

## 4. Расчёт ФОТ

`backend/app/services/payroll.py::calculate_payroll_for_day`

1. Удаляет старые `PayrollEntry` (не скорректированные вручную) и `FotSummary` за день/отчёт — защита от дублей при повторном пересчёте
2. Берёт **только закрытые** смены (`closed_at IS NOT NULL`) за день и филиал
3. Для каждой смены:
   - `total_minutes` — из поля или вычисляется из таймстампов; если `<= 0` — смена пропускается (не оплачивается)
   - находит активную ставку сотрудника на дату (`employee_rates`, по `effective_from`/`date_to`)
   - **billing_minutes** = `approved_hours * 60`, если задано вручную (менеджером/бухгалтером) и `> 0`, иначе `total_minutes`
   - **Формула оплаты**:
     - `fixed_daily` (фиксированная ставка): `base_pay = fixed_daily_rate or rate`, бонус = 0
     - `hourly` (почасовая): `base_pay = (rate / 60) * billing_minutes`
   - **Бонус кассира-администратора**: начисляется только если: должность категории `admin`, сотрудник == кассир, закрывавший день, и бонус ещё не выдавался за день. Формула — `cashier_bonus(orders_count, work_date)` = `orders_count * (7₽ Пн-Чт+Вс / 5₽ Пт-Сб)`
   - `total_pay = base_pay + bonus`
4. Создаёт `PayrollEntry` на каждого сотрудника
5. Суммирует по категориям должностей (`category_fot`: kitchen/admin/tech/courier/reserve), создаёт `FotSummary`:
   - `total_fot_pct = total_fot / revenue * 100`
   - `kitchen_fot_pct = kitchen_fot / revenue * 100`
   - **Статусы** (вычисляются через `app/business_rules.py`, единый источник после рефакторинга от 2026-06-08):
     - `total`: `< 27.5%` → green, `<= 29%` → yellow, иначе → red
     - `kitchen`: `< 14.5%` → green, `<= 15.5%` → yellow, иначе → red

> ⚠️ `takeaway_count` собирается по всей цепочке (отчёты, аналитика), но **не участвует** в формуле расчёта ФОТ — информационное поле.

## 5. Основные ограничения системы

1. **`employee` не имеет прямого доступа к API через роль User** — только PIN-флоу или отдельный кабинет с параллельной системой JWT-токенов (`type=employee`)
2. **`/shifts/live` и `/branches`** — намеренно публичные эндпоинты без авторизации (для табло мониторинга и форм выбора филиала); с 2026-06-08 ФИО в `/shifts/live` маскируются (`"Иван И."` вместо полного имени) — см. `CHANGELOG.md`
3. **Пересчёт ФОТ полностью перезаписывает день** — каждый вызов `calculate_payroll_for_day` удаляет несоскорректированные `PayrollEntry`/`FotSummary` и создаёт заново; ручные правки (`is_corrected=True`) сохраняются
4. **Бонус кассира выдаётся не более одного раза за день**, и только сотруднику с должностью категории `admin`, который закрыл день как кассир
5. **Доп. смену (вторую за день) может открыть только кассир** — обычный сотрудник не может открыть вторую смену через `/shifts/open`
6. **`test_mode.py` (`/test`)** работает только при `ENVIRONMENT=development|staging` и жёстко привязан к `branch_id=1` — должен быть отключён в проде (`ENVIRONMENT=production`)
7. **Нет встроенного покрытия тестами** для `calculate_payroll_for_day` — критическая бизнес-логика проверяется вручную через `test_mode.py`
8. **Известные архитектурные слабости** подробно описаны в `PROJECT_STATUS.md` (раздел «технический долг»)
